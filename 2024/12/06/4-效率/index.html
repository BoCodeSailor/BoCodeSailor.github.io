条款16：牢记 80－20 准则

80-20 准则说的是大约 20% 的代码使用了 80% 的程序资源：

  • 大约 20% 的代码耗用了大约 80% 的运行时间；
  • 大约 20% 的代码使用了 80% 的内存；
  • 大约 20% 的代码执行 80% 的磁盘访问；
  • 大约 80% 的维护投入在 20% 的代码上。

那怎么找出这 20% 的代码来进行优化呢？可以通过 Profiler（翻译为分析器） 来分析内存、占用 CPU、启动时间、网络情况、功耗等各个指标，而不要凭感觉或经验来判断。

条款17：考虑使用 lazy evaluation（懒惰计算法）

“考虑使用 lazy evaluation（懒惰计算法）” 强调了在编程中通过推迟计算（即懒惰计算）来提高程序的性能，减少不必要的计算开销。懒惰计算指的是，只有在结果实际需要时才进行计算，而不是一开始就执行所有操作。这种技术能够在某些情况下显著提高程序的效率，尤其是在处理大数据或计算量巨大的场景时。

懒惰计算是一种延迟计算策略，通常用于处理函数式编程或流式数据处理中。在懒惰计算中，表达式的求值被推迟，直到实际需要结果时才进行计算。

  • 即时计算（Eager Evaluation）：表达式一旦被调用就立即执行，结果会立刻计算出来。
  • 懒惰计算（Lazy Evaluation）：表达式的计算被推迟，直到需要使用结果时，计算才会发生。

懒惰计算常见的应用场景包括：

  • 链式操作：多个操作可以被串联起来，而不必每次都执行中间的计算，直到最终需要结果。
  • 流式编程：在处理大量数据流时，懒惰计算可以避免不必要的中间计算，减少资源消耗。

条款18：分期摊还期望的计算

条款18的核心是过度热情计算法（over-eager evaluation），它是在确定某些计算结果会被频繁需求时，提前计算并缓存这些结果，以避免每次请求时重复计算，从而提高效率。具体来说，过度热情计算是主动追踪和缓存那些几乎总会被需要的计算结果，而不是每次需要时才进行计算。

假设我们有一个类 DataCollection，用于存储大量的数字型数据，并提供 min(), max(), avg() 等函数来获取集合的最小值、最大值和平均值。如果我们每次调用这些函数时都重新计算最小值、最大值和平均值，尤其是数据量大的时候，会导致性能问题。

通过过度热情计算，我们可以在数据更新时缓存这些值（如最小值、最大值、平均值）。当调用 min(), max(), avg() 时，不再重新计算，而是直接返回缓存的结果。这样，虽然数据更新时需要一定的开销，但每次调用函数时，返回结果的开销大大降低。

在这个例子中，**min()**, max(), avg() 函数在第一次调用时会计算并缓存结果，以后每次调用时直接返回缓存值，从而节省了重新计算的开销。

假设你有一个企业员工管理系统，员工信息存储在数据库中，但是某些信息（如员工的隔间号）可能频繁被查询。为了避免每次都向数据库请求，可以在程序中使用缓存，存储已经查询过的隔间号。这样，如果需要查询相同员工的隔间号，可以直接从缓存中获取，而不必再次查询数据库。

条款19：理解临时对象的来源

条款19的核心是理解临时对象的来源，也就是理解什么时候会在 C++ 中生成临时对象，以及这些临时对象的成本和潜在问题。

在 C++ 中，临时对象指的是那些在程序中没有显式命名且生命周期通常很短的对象。它们并不直接出现在源代码中，而是在特定的操作或条件下由编译器生成。常见的临时对象生成场景包括：

  1. 隐式类型转换：当函数的参数类型与传入的实参类型不匹配时，编译器会生成临时对象以执行类型转换。
  2. 函数返回对象时：当函数返回一个对象时，如果这个对象没有显式命名，它就是一个临时对象。

条款20：协助完成返回值优化

条款20的核心思想是 返回值优化（Return Value Optimization, RVO），它通过消除返回对象时产生的临时对象的开销来提升程序的性能。为了理解这一条款，我们需要知道在 C++ 中，返回对象的过程如何导致性能损失，以及如何通过编译器优化避免这一损失。

当你在函数中返回一个对象时，通常会发生以下几步：

  1. 临时对象的创建： 由于 C++ 不允许直接返回局部对象的引用，返回的对象通常是通过拷贝构造或移动构造返回的，这就导致了临时对象的创建。
  2. 临时对象的销毁： 返回值被拷贝或移动到目标变量后，临时对象通常会被销毁，这也带来了额外的开销。

返回值优化（RVO）：

条款22：考虑用运算符的赋值形式取代其单独形式

条款22的核心思想是通过使用运算符的赋值形式（例如 operator+=）来替代单独形式（例如 operator+），从而提高效率并减少临时对象的开销。这个原则不仅有助于提升性能，还有助于更清晰和可维护的代码设计。下面是条款22的要点和细节分析。

背景：

在许多编程语言中，运算符重载提供了编写简洁和直观代码的能力。特别是对于用户定义类型（如 Rational 类），你可能会重载运算符来支持数学运算，比如加法和减法：

通常，程序员会希望通过 x = x + y 或 x = x - y 这样的代码来执行运算，但这些语句通常会产生临时对象。赋值形式（如 operator+=）则没有这个问题，因为它将计算结果直接写入目标对象，无需生成临时对象。

使用赋值形式的优点：

  1. 效率更高：

    • 单独形式（如 operator+）通常会返回一个新的对象，这可能会导致临时对象的构造和析构，造成性能开销。
    • 赋值形式（如 operator+=）将结果写入左操作数对象，因此不需要生成临时对象，这降低了开销。

  2. 简化实现：

总结：

  • 赋值形式运算符（如 operator+=）比单独形式运算符（如 operator+）效率更高，因为赋值形式直接在左操作数上修改值，而不需要返回新对象，这避免了临时对象的构造和析构。
  • 如果你在实现这些运算符时遵循“赋值形式实现单独形式”的设计模式，你可以减少冗余代码，保证两者之间的一致性。
  • 在设计库时，你应提供两种形式的运算符，既允许客户端代码使用简洁易懂的单独形式，也允许那些更注重性能的客户端选择赋值形式运算符。
  • 返回值优化（RVO）和临时对象的成本：在实现时，尽量使用临时对象而非命名对象，以便编译器可以更好地优化返回值。

条款23：考虑变更程序库

条款23的核心思想是程序库的设计是一个折中的过程，不同的程序库会在多个方面（如效率、可扩展性、移植性、类型安全等）做出不同的权衡。选择不同的程序库会影响程序的性能特征，因此，在遇到性能瓶颈时，你可以通过替换程序库来优化性能。

关键概念

  1. 程序库的设计权衡：
    • 理想的程序库应该具备许多优秀的特性，如短小、快速、强大、灵活、可扩展、直观、普遍适用、无错误等，但在现实中，很难同时实现所有这些目标。
    • 性能 vs. 功能：优化程序库的速度和尺寸通常会导致功能的减少。例如，stdio 可能在速度上优于 iostream，但 iostream 提供了更强的功能和类型安全性。
  2. 替换程序库以优化性能：
    • I/O 性能：在处理大量 I/O 时，stdio（C 语言标准库）通常比 iostream（C++ 标准库）更快。这是因为 stdio 的实现相对较小，且通常与底层的操作系统更紧密地集成，从而提供更高的效率。
    • 通过条件编译，可以轻松地切换 stdio 或 iostream，根据性能要求来选择最佳的程序库。例如，测试代码中使用了宏 STDIO 来决定使用哪个 I/O 库。
  3. 其他程序库替换：
    • 如果程序在内存分配和释放上遇到性能瓶颈，可以考虑替换默认的 operator new 和 operator delete 实现。例如，使用更高效的内存分配库（如 jemalloc 或 tcmalloc）可能会显著提高程序的性能。
    • 性能分析：通过性能分析工具（如 gprof 或 valgrind）来找出程序的瓶颈，并考虑是否能通过替换程序库或调整算法来优化性能。

  • 速度： stdio 相对于 iostream 运行速度较快，有时可以快 20% 到 200%。

  • 程序尺寸： 使用 stdio 的程序比使用 iostream 的程序小得多，尤其是在处理大量数据时，这种差异可能非常显著。

  • stdio 优点： 较小的程序尺寸、更快的 I/O 性能。

  • iostream 优点： 提供类型安全、可扩展性和更丰富的功能。

条款24：理解虚函数、多继承、虚基类和 RTTI 所需的代价

虚函数是 C++ 中支持多态的机制之一。当你在基类中声明一个虚函数时，派生类可以重写这个函数，并通过指向基类对象的指针或引用来调用派生类的函数。

代价：

  • 内存代价：虚函数通常通过虚函数表（vtable）来实现。每个包含虚函数的类会有一个虚函数表，这个表存储了指向该类所有虚函数的指针。每个对象通常会有一个指向其类虚函数表的指针（vptr）。因此，虚函数带来了每个对象多一个指针的内存开销。
  • 性能代价：调用虚函数时，需要通过虚函数表来查找目标函数的地址。这是一个间接调用，会比直接函数调用慢，尤其是当虚函数层级较深时，代价会更大。

影响：

  • 虚函数带来了灵活的多态机制，但需要谨慎考虑性能要求。特别是对于频繁调用的虚函数，性能代价可能会比较显著。

多继承的代价

C++ 支持多继承，即一个类可以继承多个基类。多继承使得类可以从多个基类中继承行为，但也带来了设计上的复杂性。

代价：

  • 内存代价：每个基类的子对象需要包含一个完整的副本。换句话说，当一个类从多个基类继承时，每个基类的成员都会被复制到派生类的对象中，导致对象的内存占用增大。如果多个基类中有相同的成员变量或相同的虚函数表，这会导致不必要的内存浪费。
  • 性能代价：多继承可能会使得函数调用变得更加复杂。尤其是在访问继承自多个基类的成员时，编译器可能需要通过多个偏移量来访问每个基类的成员，这会导致访问速度变慢。
  • 复杂性：多继承可能导致类的设计变得复杂，继承关系和成员访问可能变得不清晰，增加了维护难度。

虚基类（Virtual Base Classes）的代价

虚基类是在多继承中处理“钻石问题”时使用的一种技术。当多个派生类继承自同一个基类时，虚基类确保该基类只会在最终派生类中存在一份副本。

代价：

  • 内存代价：虚基类通常会引入一个额外的指针（通常是指向虚基类的指针）来确保多重继承时只有一份基类子对象。每个虚基类都会在每个对象中占用一个指针，这增加了内存使用。
  • 性能代价：访问虚基类的成员可能需要通过额外的指针间接访问，这使得访问速度变慢。

影响：

  • 虚基类解决了多继承中的继承冲突问题，但在性能和内存占用上增加了代价。因此，设计时需要考虑是否真的需要虚基类来解决钻石继承问题，避免不必要的复杂性。

运行时类型信息（RTTI）的代价

RTTI 是 C++ 提供的一种机制，用于在程序运行时获取对象的类型信息。它通常与 dynamic_cast 和 typeid 操作符一起使用，用于类型转换和类型检查。

代价：

  • 内存代价：RTTI 需要在每个类的对象中存储额外的类型信息。例如，每个包含虚函数的类通常都包含一个指向该类类型信息的指针。这会增加每个对象的内存开销。
  • 性能代价：使用 dynamic_cast 或 typeid 时，程序需要进行运行时的类型检查，通常涉及到查找对象的类型信息。这是一个额外的开销，尤其是在频繁进行类型转换和类型检查时，性能损失会更加明显。

影响：

  • RTTI 提供了强大的动态类型识别能力，但它也会增加内存和性能开销。对于要求极高性能的程序，应该谨慎使用 RTTI。
