条款25：将构造函数和非成员函数虚拟化

在 C++ 中，虽然构造函数和非成员函数不能直接声明为虚拟函数，但有一些技术和设计模式可以模拟这些行为。这条款介绍了如何在设计中通过虚拟化构造函数和非成员函数来提高灵活性和可扩展性，特别是在涉及到基类和派生类的多态性时。

虚拟构造函数（Virtual Constructors）

构造函数是用于初始化对象的特殊成员函数，它们不能直接声明为虚拟的，因为在对象的构造过程中，虚拟机制（即虚函数表）尚未建立，无法执行多态性调用。然而，可以通过设计一个“工厂函数”来模拟虚拟构造函数的效果。该函数根据输入的不同创建不同类型的对象，动态决定要构造的具体类型。

虚拟拷贝构造函数

虚拟拷贝构造函数是一种特殊的设计模式，允许基类定义一个虚拟的 clone 函数，派生类通过覆盖 clone 函数来实现多态的拷贝构造行为。通过这种方式，可以通过基类的指针或引用进行对象的复制，而不需要显式地知道对象的具体类型。

条款26：限制某个类所能产生的对象数量

在一些场景下，我们可能需要限制某个类的对象实例化数量。比如，在系统中可能只有一个打印机对象，或者资源有限，像文件描述符等的数量受到限制。为了实现这一目标，我们可以通过几种技术来控制和限制某个类所能创建的对象数量。

禁止构造函数的直接调用

最直接的方法就是通过控制类的构造函数，使得外部无法直接创建该类的对象。这可以通过将构造函数声明为 private 或 delete 来实现。

继承和包含的问题

当尝试继承或包含一个禁止在堆中产生对象的类时，会遇到问题。为了解决继承问题，可以将基类的构造函数或析构函数声明为受保护（protected）。对于包含问题，可以将包含的对象改为指向对象的指针，并在需要时使用new和delete或destroy方法。

判断对象是否在堆中

没有通用且有效的方法可以区分堆和栈对象。一些策略，如重载operator new或使用地址空间的布局特性来判断，都存在问题，不是可移植的解决方案。因此，通常不推荐尝试判断对象是否在堆中，而是应该设计代码以避免这种需求。

条款28：智能（smart）指针

智能指针是一种外观和行为都被设计成与内建指针相类似的对象，不过它能提供更多的功能。它们有许多应用的领域，包括资源管理（参见条款 M9、M10、M25 和 M31）和重复代码任务的自动化（参见条款 M17 和 M29）。

条款29：引用计数

引用计数的动机

  1. 简化内存管理：引用计数可以简化堆对象的簿记工作，便于管理内存。当一个对象不再被任何引用所指向时，它会自动销毁自己，从而实现简单的垃圾回收体系。

  2. 节省空间和提升效率：如果很多对象有相同的值，通过引用计数可以让这些对象共享同一个值的实现，从而节省内存并提升效率。

实现引用计数

在实现引用计数时，需要一个结构来存储引用计数和对象的值。这个结构通常被嵌套在管理它的类内部，以保持值和引用计数的关联。例如，String类内部可以有一个StringValue结构，用于存储引用计数和字符串数据。

StringValue的构造函数初始化引用计数为1，并且复制传入的字符串值。

管理引用计数

  • 拷贝构造函数和赋值操作符：在拷贝构造函数和赋值操作符中，需要增加和减少引用计数。当引用计数减到0时，需要删除对应的值。
  • 写时复制（Copy-on-Write）：对于可能改变对象的操作（如operator[]），需要先检查是否需要复制值，以确保对共享资源的修改不会影响到其他对象。

引用计数的基类

可以创建一个基类RCObject来管理引用计数，使得任何需要引用计数的类都可以继承自这个基类。

自动操作引用计数

可以通过智能指针类RCPtr来自动管理引用计数，减少在每个类中手动管理引用计数的负担。

条款30：Proxy Classes（替身类、代理类）

这个tips建议看书，细节繁多，这里只写了设计模式中的代理模式

在软件设计中，代理类（Proxy class）是一种设计模式，用于通过代理对象控制对其他对象的访问。代理类通常会在实际对象和客户端之间充当中介，从而增加额外的功能，如延迟加载、安全控制、访问控制、日志记录等。

代理类的核心思想是通过为目标对象提供代理对象来进行访问控制。代理类通常遵循“和目标对象相同的接口”，因此它能够像目标对象一样被使用。通过代理类，客户端可以像调用目标对象一样调用代理对象，而代理对象可以在内部决定是否直接调用目标对象的真实方法。

代理模式的几种常见类型

  1. 虚拟代理（Virtual Proxy）： 用于延迟对象的创建和加载。代理对象在实际需要时才会创建或加载目标对象。这种方式常用于占用资源较大的对象，如图像、数据库连接等。
  2. 保护代理（Protective Proxy）： 用于控制访问权限。代理对象可以检查客户端是否有权限访问目标对象。如果没有权限，代理对象可以拒绝访问或进行其他处理。
  3. 远程代理（Remote Proxy）： 用于访问位于不同地址空间（如不同的进程、机器）的对象。通过网络协议，代理对象将请求发送到远程对象，通常用于分布式系统中。
  4. 智能代理（Smart Proxy）： 代理对象提供额外的功能，如引用计数、内存管理、日志记录等。智能代理类通常会在访问目标对象时执行一些额外的处理。

代理类的优势和劣势

优势：

  • 解耦：代理类通过控制对目标对象的访问，客户端与目标对象解耦，简化了对象的使用方式。
  • 增强功能：可以为目标对象添加额外的功能，如延迟加载、权限控制等，而不需要修改目标对象的代码。
  • 性能优化：通过延迟加载、缓存等手段，提高系统的性能和响应速度。

劣势：

  • 增加复杂性：代理类增加了额外的间接层，可能导致系统变得更加复杂，难以维护。
  • 性能开销：虽然代理类能够优化性能，但在某些情况下，代理对象本身的间接调用可能会带来额外的性能开销。

条款31：让函数根据一个以上的对象类型来决定怎么虚拟

C++ 中的虚函数机制支持 单重分派（single-dispatch），即通过一个对象来决定调用哪个函数。它不能直接支持基于多个对象类型的 双重分派（double-dispatch）或 多重分派（multiple-dispatch）。不过，我们可以通过设计模式和一些技巧来模拟双重分派和多重分派，以便根据多个对象的类型来决定函数的行为。

在这一条款中，我们讨论了如何处理多个对象类型的碰撞检测问题。假设你有一个游戏场景，其中包含不同的游戏对象（例如飞船、太空站和小行星），这些对象之间可能会发生碰撞，而碰撞的处理逻辑是基于对象类型的组合来决定的。

解决方案：使用非成员函数+ 运行时类型识别（RTTI）

我们通过以下步骤来实现：

  1. 定义基类和派生类：基类 GameObject 和其派生类 SpaceShip、SpaceStation、Asteroid。
  2. 定义碰撞处理函数：对于每对对象类型的组合，定义一个处理碰撞的函数。
  3. 碰撞处理的映射表：使用一个碰撞处理映射表来存储每对对象类型的碰撞处理函数，并通过 RTTI（运行时类型识别）来决定调用哪个碰撞函数。
