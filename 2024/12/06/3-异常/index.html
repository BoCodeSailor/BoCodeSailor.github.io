异常

条款 9:利用 destructors 避免泄露资源

析构函数的主要特点使其成为避免资源泄露的重要工具：

  1. 自动调用：对象的生命周期结束时，析构函数会自动被调用，无需显式调用。
  2. 与对象绑定：资源与对象的生命周期绑定，当对象销毁时，相关资源也会被释放。

资源泄露的常见场景

  1. 手动分配内存未释放

  2. 打开的文件未关闭

  3. 未释放锁或网络连接

利用析构函数管理资源

通过在类中封装资源，并在析构函数中释放资源，可以有效避免泄露。这一思想称为 资源获取即初始化（RAII，Resource Acquisition Is Initialization）。

示例：管理动态内存

示例：管理文件句柄

示例：管理互斥锁

RAII 的关键思想

  1. 将资源的生命周期绑定到对象的生命周期
    • 使用构造函数初始化资源。
    • 在析构函数中释放资源。
  2. 异常安全
    • 即使程序抛出异常，析构函数也会自动被调用，确保资源被正确释放。
    • RAII 使得代码在异常处理方面更具健壮性。

现代 C++ 中的 RAII 工具

现代 C++ 提供了大量基于 RAII 的标准库工具，可以简化资源管理：

  1. std::unique_ptr 和 std::shared_ptr

    • 智能指针用于管理动态内存，避免手动调用 delete。

  2. std::lock_guard 和 std::unique_lock

    • 自动管理锁，确保异常发生时锁也会被释放。

  3. std::fstream

    • 文件流类自动管理文件句柄。

条款 10 :在 constructors 内阻止资源泄露

在 C++ 中，构造函数用于初始化对象。如果在构造函数内分配了资源（如动态内存、文件句柄等），但构造函数在完成之前抛出了异常，这些资源可能无法被正确释放，从而导致资源泄露。

问题：

  • 如果在 fopen 失败后抛出异常，析构函数不会被调用，因为对象未完全构造。
  • 这导致 data 分配的内存永远不会释放，造成内存泄露。

如何在构造函数中阻止资源泄露

RAII：将资源封装到独立类

将每种资源（如内存、文件句柄）封装到专门的 RAII 类中，这样即使构造函数抛出异常，封装资源的对象析构函数也会自动释放资源。

优势：

  • data 的内存由 std::unique_ptr 管理，异常时自动释放。
  • 文件句柄由 FileHandle 管理，异常时自动关闭。

使用智能指针管理动态内存

动态内存管理的常见问题可以通过智能指针（如 std::unique_ptr 或 std::shared_ptr）来解决。

  • 如果构造函数抛出异常，std::unique_ptr 和 std::shared_ptr 会自动释放已分配的资源。

避免在构造函数中进行复杂逻辑

构造函数中的复杂逻辑可能增加异常抛出的可能性。将复杂逻辑分离到其他方法中，有助于降低构造函数中资源管理的复杂性。

条款 11: 禁止异常流出 destructors之外

  • 如果析构函数在异常传播期间本身再抛出一个异常，这会导致双重异常。
  • 双重异常在 C++ 中会调用 std::terminate，导致程序非正常终止。

流程：

  • throw std::runtime_error("Outer exception") 开始传播异常。
  • 在 Example 的析构函数中，再次抛出异常。
  • 两个异常冲突，程序调用 std::terminate，直接崩溃。

如何避免析构函数中传播异常？

如果析构函数中可能会发生异常，应捕获它们并妥善处理，确保异常不会传播。

要点：

  • 捕获所有异常（通常使用 catch (...)）。
  • 记录或处理异常，避免其传播。

条款12：了解 “抛出一个 exception ” 与 “传递一个参数” 或 “调用一个虚函数”之间的差异

异常处理类型转换

在C++中，异常处理确实只支持有限的类型转换。当抛出一个异常时，catch块会尝试匹配异常对象的类型。只有两种情况下的类型转换是允许的：

  1. 从派生类到基类的转换：如果抛出了一个派生类类型的异常，可以用其基类类型的引用或指针来捕获。
  2. void* 类型转换：可以使用 const void* 捕捉任何指针类型的异常，但这不是推荐的做法，因为它丧失了类型安全性。

重要的是，如果抛出了一个 int 类型的异常，你不能用 double 类用来接收，因为它们之间没有继承关系，也不是指针类型。此外，C++的异常处理机制不支持隐式的数值类型转换。

异常处理中的 catch 子句顺序

关于多个 catch 子句，它们确实是按照出现的顺序进行匹配的。因此，更具体的异常类型应该放在更通用的异常类型之前。例如，如果你有一个 runtime_error 和一个更通用的 exception，你应该把 runtime_error 放在前面，否则它将永远不会被匹配到，因为所有 runtime_error 都是 exception 的子类

条款14：明智运用异常规范（exception specifications）

异常规范是C++中一个特性，允许开发者指定函数可能抛出的异常类型。然而，在实际应用中，异常规范往往被证明是有问题的，并且在现代C++编程中并不推荐使用。

这告诉编译器 func 函数只会抛出 int 或者 std::bad_alloc 类型的异常。如果 func 抛出了其他类型的异常，那么程序将调用 std::unexpected 函数，其默认行为是终止程序。

为什么异常规范不被推荐

  1. 性能影响：编译器为了确保函数只抛出声明的异常类型，可能会插入额外的代码来进行检查，这可能导致性能下降。
  2. 维护困难：当函数内部实现发生变化时，你必须记得更新异常规范，否则可能会导致未捕获异常的行为。
  3. 违反封装性：异常规范暴露了函数内部可能抛出的异常，这实际上破坏了函数的封装性，因为它泄露了实现细节。
  4. 缺乏灵活性：如果一个函数调用了另一个具有更广泛异常规范的函数，或者抛出了新的异常类型，那么你就不得不修改它的异常规范。
  5. 编译器支持不一致：不同编译器对异常规范的支持和优化程度不一样，这可能导致跨平台开发中的问题。

自C++11起，标准委员会意识到这些缺点，并引入了 noexcept 规范来替代旧的异常规范语法。noexcept 更加简单明确，它只是表示一个函数不会抛出任何异常。对于那些确实不会抛出异常的函数，使用 noexcept 可以帮助编译器进行优化，并且可以提高某些操作（如移动语义）的安全性和效率。

条款15：了解异常处理的成本

运行时性能开销

  • 堆栈展开：当一个异常被抛出时，程序会执行所谓的“堆栈展开”过程。这意味着从抛出异常的点开始，逐层返回到调用链中的每一层，直到找到匹配的 catch 块。这个过程中，每个函数的局部对象都会被销毁，这可能涉及非平凡的析构函数调用。
  • 资源管理：为了确保在异常发生时正确地释放资源（如内存、文件句柄等），通常需要使用RAII（Resource Acquisition Is Initialization）原则。这意味着为每种资源创建一个类，并在该类的构造函数中获取资源，在析构函数中释放资源。虽然这是个好习惯，但也会增加额外的开销。

编译时成本

  • 代码膨胀：编译器必须生成额外的代码来支持异常处理，包括设置和清理异常处理信息。这可能导致二进制文件变大，并增加了编译时间。
  • 优化限制：由于异常的存在，编译器在某些情况下无法进行特定的优化。例如，如果一个函数声明为可以抛出异常，那么编译器就不能假设它总是成功完成，这可能会限制内联和其他优化策略
